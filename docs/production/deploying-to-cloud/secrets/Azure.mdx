### YAML

To use secrets in YAML files (e.g. in data integration pipelines):

1. Add the following keys and values to your environment variables so that
Mage can access the Azure Key Vault:
    1. `AZURE_KEY_VAULT_URL`: The Azure Key Vault URI.
    1. `AZURE_CLIENT_ID`: Azure Active Directory [client ID](https://learn.microsoft.com/en-us/azure/active-directory/develop/msal-client-application-configuration#client-id).
    1. `AZURE_CLIENT_SECRET`: Azure Active Directory [client secret](https://learn.microsoft.com/en-us/azure/active-directory/develop/quickstart-register-app#add-a-client-secret).
    1. `AZURE_TENANT_ID`: Azure Active Directory [tenant ID](https://learn.microsoft.com/en-us/azure/azure-portal/get-subscription-tenant-id#find-your-azure-ad-tenant).
1. Store a secret in Azure Key Vault.
1. Use the following syntax in your YAML file to interpolate secret values from Azure Key Vault:
    ```yaml
    "{{ azure_secret_var('secret_name') }}"
    ```

    For example:
    ```yaml
    api_key: "{{ azure_secret_var('API_KEY') }}"
    access_token: "{{ azure_secret_var('ACCESS_TOKEN') }}"
    ```

### Python

Here is the example python code to get secrets from Azure Key Vault:

```python
from mage_ai.services.azure.key_vault.key_vault import get_secret

get_secret('secret_name')
```

## Using multiple Azure Key Vaults

A single Mage instance can retrieve secrets from any number of Azure Key
Vaults. There are three ways to target a specific vault:

1. **Vault profiles** (recommended) — configure once, reuse by name
2. **Explicit vault URL** — pass the URL directly in each call
3. **Pipeline variables** — resolve the vault URL dynamically at runtime

### Vault profiles (recommended)

Profiles let you configure vault credentials **once** via environment variables
and reference them by a short name in your pipelines. This avoids entering
`client_id`, `client_secret`, and `tenant_id` every time.

#### Step 1: Set environment variables

For each vault, define environment variables following this naming convention:

```bash
# Customer 1 vault
AZURE_KEY_VAULT_PROFILE_CUSTOMER1_URL=https://customer1.vault.azure.net/
AZURE_KEY_VAULT_PROFILE_CUSTOMER1_CLIENT_ID=<client-id>
AZURE_KEY_VAULT_PROFILE_CUSTOMER1_CLIENT_SECRET=<client-secret>
AZURE_KEY_VAULT_PROFILE_CUSTOMER1_TENANT_ID=<tenant-id>

# Customer 2 vault
AZURE_KEY_VAULT_PROFILE_CUSTOMER2_URL=https://customer2.vault.azure.net/
AZURE_KEY_VAULT_PROFILE_CUSTOMER2_CLIENT_ID=<client-id>
AZURE_KEY_VAULT_PROFILE_CUSTOMER2_CLIENT_SECRET=<client-secret>
AZURE_KEY_VAULT_PROFILE_CUSTOMER2_TENANT_ID=<tenant-id>
```

The pattern is `AZURE_KEY_VAULT_PROFILE_<NAME>_<FIELD>` where `<NAME>` is your
chosen profile name (case-insensitive) and `<FIELD>` is one of `URL`,
`CLIENT_ID`, `CLIENT_SECRET`, or `TENANT_ID`.

<Note>
  Only the `URL` field is required. If `CLIENT_ID`, `CLIENT_SECRET`, and
  `TENANT_ID` are omitted, `DefaultAzureCredential` is used for that profile
  (e.g. when the vault is accessible via managed identity).
</Note>

#### Step 2: Use the profile in your pipeline

**YAML:**
```yaml
api_key: "{{ azure_secret_var('API_KEY', profile='customer1') }}"
db_password: "{{ azure_secret_var('DB_PASS', profile='customer2') }}"
```

**Python:**
```python
from mage_ai.services.azure.key_vault.key_vault import get_secret

get_secret('API_KEY', profile='customer1')
get_secret('DB_PASS', profile='customer2')
```

That's it — no credentials in your pipeline code, no repeating `client_id` /
`client_secret` / `tenant_id` in every call.

#### Docker / Docker Compose example

```yaml
services:
  mage:
    image: mageai/mageai:latest
    environment:
      # Default vault (backward compatible)
      AZURE_KEY_VAULT_URL: https://default.vault.azure.net/
      AZURE_CLIENT_ID: ${AZURE_CLIENT_ID}
      AZURE_CLIENT_SECRET: ${AZURE_CLIENT_SECRET}
      AZURE_TENANT_ID: ${AZURE_TENANT_ID}
      # Customer 1 profile
      AZURE_KEY_VAULT_PROFILE_CUSTOMER1_URL: https://customer1.vault.azure.net/
      AZURE_KEY_VAULT_PROFILE_CUSTOMER1_CLIENT_ID: ${CUSTOMER1_CLIENT_ID}
      AZURE_KEY_VAULT_PROFILE_CUSTOMER1_CLIENT_SECRET: ${CUSTOMER1_CLIENT_SECRET}
      AZURE_KEY_VAULT_PROFILE_CUSTOMER1_TENANT_ID: ${CUSTOMER1_TENANT_ID}
      # Customer 2 profile
      AZURE_KEY_VAULT_PROFILE_CUSTOMER2_URL: https://customer2.vault.azure.net/
      AZURE_KEY_VAULT_PROFILE_CUSTOMER2_CLIENT_ID: ${CUSTOMER2_CLIENT_ID}
      AZURE_KEY_VAULT_PROFILE_CUSTOMER2_CLIENT_SECRET: ${CUSTOMER2_CLIENT_SECRET}
      AZURE_KEY_VAULT_PROFILE_CUSTOMER2_TENANT_ID: ${CUSTOMER2_TENANT_ID}
```

### YAML — explicit vault URL

```yaml
api_key: "{{ azure_secret_var('API_KEY', vault_url='https://my-other-vault.vault.azure.net/') }}"
```

### YAML — vault URL from a pipeline variable

When the same pipeline is triggered once per customer, pass the vault URL as a
runtime variable and reference it with `variables()`:

```yaml
api_key: "{{ azure_secret_var('API_KEY', vault_url=variables('customer_vault_url')) }}"
```

Then supply `customer_vault_url` via the trigger or API call that starts
the pipeline run.

### YAML — per-vault credentials (inline)

If you cannot use profiles, you can provide credentials explicitly in each
call:

```yaml
api_key: "{{ azure_secret_var('API_KEY', vault_url='https://customer1.vault.azure.net/', client_id='...', client_secret='...', tenant_id='...') }}"
```

When credentials are omitted, `DefaultAzureCredential` is used (same behavior
as before).

<Warning>
  All three credential parameters (`client_id`, `client_secret`, `tenant_id`)
  must be provided together. Passing only some of them will raise an error.
  Avoid hardcoding credentials in YAML files — prefer vault profiles or
  pipeline variables instead.
</Warning>

### Python

```python
from mage_ai.services.azure.key_vault.key_vault import get_secret

# Default vault (AZURE_KEY_VAULT_URL env var)
get_secret('secret_name')

# Using a named profile (recommended for multi-vault)
get_secret('secret_name', profile='customer1')

# Explicit vault URL — uses DefaultAzureCredential
get_secret('secret_name', vault_url='https://customer1.vault.azure.net/')

# Explicit vault URL with dedicated service principal
get_secret(
    'secret_name',
    vault_url='https://customer2.vault.azure.net/',
    client_id='...',
    client_secret='...',
    tenant_id='...',
)
```

<Note>
  `SecretClient` instances are cached per unique combination of vault URL and
  credentials, so repeated calls to the same vault reuse the existing
  connection.
</Note>
